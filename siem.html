SIEM & Logs

Structural anomaly signals
for every log line.

UMI applies to network syslogs and software events with zero schema changes. The same 128-bit envelope, the same SQL query, the same compliance trace structure used for AI outputs. UMI adds a queryable structural signal layer that complements existing rule-based SIEM detection.

Scope

UMI adds a structural metadata layer to log analysis. It does not replace existing SIEM platforms, regex rule engines, or domain-specific threat intelligence. It produces a queryable anomaly signal — interpretation and response remain with qualified security teams.

Why it matters

Three things you cannot do with raw logs alone

Security Analyst
SSH brute force at 03:14 UTC.
You arrive at 09:00.
Was it part of a coordinated attack?
With raw logs: hours of grepping and correlating. With UMI: one SQL query filtering A4_CLARITY below threshold in the 6-hour window. The answer is ready before your second coffee.
Compliance & Risk Officer
NIS2 gives you 72 hours.
The regulator wants to see that you detected it when it happened — not retrospectively.
A UMI trace is the forensic receipt. It documents that the anomaly was detected and classified at the moment it occurred, with a tamper-evident checksum. That is your 72-hour evidence chain.
SIEM Architect
Your SIEM ingests terabytes daily.
You pay per GB.
Most of it is routine PASS-level traffic.
UMI as a pre-processing layer means only anomalous records need forwarding to your central SIEM. Routine events are encoded locally as 16-byte traces. Ingestion costs drop by an order of magnitude.
NIS2 forensic query — find every anomaly in the incident windowSQL
-- 03:14 UTC incident. What else happened in the same window? SELECT trace_id, timestamp_utc, semantic_summary->>'domain' AS subsystem, (semantic_summary->>'coherence_score')::float AS a4 FROM umi_traces WHERE (semantic_summary->>'coherence_score')::float < 0.25 AND timestamp_utc BETWEEN '2026-03-24T02:00:00Z' AND '2026-03-24T05:00:00Z' ORDER BY a4 ASC; -- Returns every critical anomaly ranked by severity. -- This IS your NIS2 Art.23 forensic timeline.

Complementary approach

UMI alongside rule-based detection

Existing SIEM platforms use rule-based detection. UMI adds a structural signal layer. The two approaches address different failure modes.

Rule-based SIEM limitations
Rules written manually for each known attack pattern
Novel variants can evade rigid pattern matching
Separate rule sets for each log format and source
Alert fatigue from high false-positive rates on noisy rules
Rules do not produce structured compliance records
What UMI adds on top
A4_CLARITY measures deviation from baseline execution graph
Structural deviation measurable independent of specific pattern
Same schema for all log types, sources, and substrates
Single tunable A4_CLARITY threshold instead of thousands of rules
Every record is a SQL-queryable compliance artifact
The key insight

A conformant encoder reads log text and extracts structural properties. A4_CLARITY in the log context measures deviation from baseline — the same field, the same threshold, the same routing logic used for AI output analysis. The schema does not change. Not one field.

API — available now
You do not need to wait for edge hardware. The UMI encoding API is available today — submit log text, receive a structured 16-byte trace with canonical checksum, store it alongside your raw log. The cost reduction and compliance benefits apply immediately via API, with no infrastructure changes required.
